### 2. 現有 Collection / Plant 結構盤點報告

**A. 欄位總表**

| 欄位名稱 | 用途 | 實際值示例 | 是否穩定 | 可否前台列表使用 |
| :--- | :--- | :--- | :--- | :--- |
| `id`/`serialNumber` | 唯一識別與履歷號 | `UNKNOWN-ID` 或 `9a8b...` | 是 | 可 (副標題) |
| `name`/`localName` | 稱呼 | 「大粒龜甲」 | 是 | 可 (主標題) |
| `status` | 販售與申請狀態 | `exhibiting`, `sold`, `培育中` | 否 | 是 (需 Mapping 轉換) |
| `category` | 屬系/分類 | `dioscorea` | 是 | 可 (篩選參數) |
| `grade` | 評級 | `S`, `A`, `B`, `Starter` | 是 | 可 (卡片標籤) |
| `size` | 尺寸資訊 | `15cm` | 是 | 可 (次要資訊) |
| `stats.*` | 雷達圖評價 | 表現、平衡、比例評價字串 | 是 | 否 (字多，僅詳細頁) |
| `images` / `cover` | 圖片/主圖 | `https://...` | 是 | 可 (卡片主圖) |

**B. 對 Collection 架構最有影響的 3 個發現**
1. **Status 欄位維度混亂**：業務狀態 (販售/已售) 與培育階段 (培育中) 混用。
2. **缺乏多維篩選**：目前僅有分類與狀態，未來數量增長時，缺少依尺寸或等級篩選會難以尋找。
3. **評測維度文字化**：評價皆為字串描述而非分數，無法進行排行與分數篩選。

### 3. FAQ 現況整理報告

**A. 重點盤點結果**
- **偏向養護百科**：目前 FAQ 幾乎全是「如何照顧、如何判斷休眠」等植物知識，而非客服支援 (Service Support)。
- **缺失商務守則**：沒有提及會員申請流程、等待時間、付款方式、運送條件、死亡補償等交易細節。
- **系統說明錯位**：解釋 `/collection` 標籤意義的題目被錯置在 FAQ 中，逛列表的使用者無法第一時間看到。

---

### 4. 會員／申請／追蹤流程盤點報告

**A. 流程總覽**
1. **未登入**：瀏覽 -> 點申請/追蹤 -> 強制跳轉 `/login` -> 進入 `/member` 落入死胡同。
2. **已登入 (Pending)**：顯示黃色待審核 Banner，申請/追蹤功能被 Alert 阻擋。
3. **已登入 (Approved)**：可追蹤 Observation 狀態植物，可申請 Available 狀態植物，能從儀表板監看紀錄。

**B. 流程死胡同 (Dead End) 與斷點**
- **登入中斷**：登入完成後直接進入 Member 頁，不會將使用者帶回稍早瀏覽的植物頁面。
- **Favorites 死路**：已追蹤的植物若售出 (Archived)，點進去無法取消追蹤，卡在清單中。
- **申請斷皮**：申請單 Approved 後，流程隨之中斷，缺乏線上付款與出貨的資訊閉環。

---

### 5. 表單與客服入口盤點報告

**A. 總結與建議**
- **建議合併**：General Inquiry (`/about`) 與 Business Inquiry (`/business`) 欄位雷同，應合併為統一的 `/contact` 分裝下拉選項。
- **處理流向盲點**：使用者提交 Inquiry 後，前台無留單紀錄可查；後台 Admin 改為 closed 狀態時，使用者也無從得知。
- **無防護測試**：表單缺乏防機器人機制 (如 reCAPTCHA)，易造成後台資料庫被灌爆。

---

### 6. 內容更新需求盤點報告

**A. 維護成本評估**
- **高頻重裝備區塊**：`/collection` 與 `/admin/applications` 極度依賴 Admin 人工作業（建立履歷、處理申請、更動狀態）。
- **高風險閒置區塊**：`/posts` 系統若無充沛人力定期產出內容，容易變成無人更新的廢墟。
- **低頻穩定區塊**：Home / About / Learn / Membership 初始化後即可長久運作。

## ▍第二部分：營運素材與新架構遷移盤點

### 任務 1｜Selection / Archive 可用素材盤點

**A. 統計數據**
- 可用狀態：Available (1 筆)、Observation (1 筆)、Archived (1 筆)。均為測試資料。
- 圖片完整度：**0 筆**具備 >4 張照片。目前每一筆僅提供 1 張封面圖。
- 欄位缺失：Grade 完全留白，Serial Number 現依賴系統 ID，Size 無統一數值格式。

**B. 影像缺口結論**
目前 **100% 缺乏**「裂紋近拍、芽點近拍、側光立體起伏、季節狀態比較、環境尺度參照」。在啟動新版頁面之前，必須完全重啟商品拍攝計畫。

---

### 任務 2｜Membership 真實審核流程與問卷需求盤點

**A. 目前缺失**
Google OAuth 僅取得姓名與 Email。要做到「經驗背景審核」，Admin 實際上毫無數據可參考。

**B. 建議新增的 `/membership/apply` 問卷欄位**
1. 養護年資 (不到1年 / 1-3年 / 3年以上)
2. 環境描述 (室內植物燈 / 陽台半日照 / 露台全日照)
3. 養護數量 (純數字)
4. 環境照片提供 (上傳區塊)
5. 申請動機與尋找型態 (文字框)

---

### 任務 3｜交易閉環與售後規則盤點

**A. 缺失的後半段旅程**
目前流程停在 Admin 點擊 Approved。後續欠缺的必備節點：
- **Quoted (報價通知)**：提供匯款帳戶或 Stripe 結帳連結。
- **Paid (已付款)**：使用者回報對帳機制。
- **Preparing (包裝理貨)**：脫盆風乾或含盆準備的提示。
- **Shipped (已出貨)**：提供物流單號查詢。

**B. 建議新建模組**
需在前台新增 `/account/applications/:id` 詳細訂單頁，補上對話通訊區、匯款回報表單以及售後指南下載按鈕。

---

### 任務 4｜內容供稿能力與更新節奏盤點

**A. 策略建議**
- **靜態化**：`Standards` 與 `Cultivation` 屬於萬年不變的核心準則，應作為開發時的一次性靜態母頁。
- **緩兵之計**：`Journal` (原 Posts) 發文成本極高，若團隊無法保證每月產出，強烈建議第一階段先行隱藏，把心力集中在 Instagram 短圖文發布。

### 任務 6｜品牌影像素材盤點

**最終結論：素材嚴重短缺。**
目前站內僅有的覆蓋圖無法支撐新版「科學、客觀、檔案化」的精神。開發初期請務必允許技術端使用合乎比例的高品質 Placeholder (佔位符) 推進版面設計。同時要求攝影組，每一株上架的 Selection 至少包含：`1張全身定裝`、`2張裂紋微距`、`1張芽點特寫`、`1張手部或空間比例尺`。
